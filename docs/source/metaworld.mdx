# Meta-World

Meta-World is a well-designed, open-source simulation benchmark for multi-task and meta reinforcement learning in continuous-control robotic manipulation. It gives researchers a shared, realistic playground to test whether algorithms can _learn many different tasks_ and _generalize quickly to new ones_ — two central challenges for real-world robotics.

- 📄 [MetaWorld paper](https://arxiv.org/pdf/1910.10897)
- 💻 [Original MetaWorld repo](https://github.com/Farama-Foundation/Metaworld)

![MetaWorld MT10 demo](https://meta-world.github.io/figures/ml45.gif)

## Why Meta-World matters

- **Diverse, realistic tasks.** Meta-World bundles a large suite of simulated manipulation tasks (50 in the MT50 suite) using everyday objects and a common tabletop Sawyer arm. This diversity exposes algorithms to a wide variety of dynamics, contacts and goal specifications while keeping a consistent control and observation structure.
- **Focus on generalization and multi-task learning.** By evaluating across task distributions that share structure but differ in goals and objects, Meta-World reveals whether an agent truly learns transferable skills rather than overfitting to a narrow task.
- **Standardized evaluation protocol.** It provides clear evaluation modes and difficulty splits, so different methods can be compared fairly across easy, medium, hard and very-hard regimes.
- **Empirical insight.** Past evaluations on Meta-World show impressive progress on some fronts, but also highlight that current multi-task and meta-RL methods still struggle with large, diverse task sets. That gap points to important research directions.

## What it enables in LeRobot

In LeRobot, you can evaluate any policy or vision-language-action (VLA) model on Meta-World tasks and get a clear success-rate measure. The integration is designed to be straightforward:

- We provide a LeRobot-ready dataset for Meta-World (MT50) on the HF Hub: `https://huggingface.co/datasets/lerobot/metaworld_mt50`.
  - This dataset is formatted for the MT50 evaluation that uses all 50 tasks (the most challenging multi-task setting).
  - MT50 gives the policy a one-hot task vector and uses fixed object/goal positions for consistency.

- Task descriptions and the exact keys required for evaluation are available in the repo/dataset — use these to ensure your policy outputs the right success signals.

## Quick start, train a SmolVLA policy on Meta-World

Example command to train a SmolVLA policy on a subset of tasks:

```bash
lerobot-train \
  --policy.type=smolvla \
  --policy.repo_id=${HF_USER}/metaworld-test \
  --policy.load_vlm_weights=true \
  --dataset.repo_id=lerobot/metaworld_mt50 \
  --env.type=metaworld \
  --env.task=assembly-v3,dial-turn-v3,handle-press-side-v3 \
  --output_dir=./outputs/ \
  --steps=100000 \
  --batch_size=4 \
  --eval.batch_size=1 \
  --eval.n_episodes=1 \
  --eval_freq=1000
```

Notes:

- `--env.task` accepts explicit task lists (comma separated) or difficulty groups (e.g., `env.task="hard"`).
- Adjust `batch_size`, `steps`, and `eval_freq` to match your compute budget.
- **Gymnasium Assertion Error**: if you encounter an error like
  `AssertionError: ['human', 'rgb_array', 'depth_array']` when running MetaWorld environments, this comes from a mismatch between MetaWorld and your Gymnasium version.
  We recommend using:

```bash
  pip install "gymnasium==1.1.0"
```

to ensure proper compatibility.

## Quick start — evaluate a trained policy

To evaluate a trained policy on the Meta-World medium difficulty split:

```bash
lerobot-eval \
  --policy.path="your-policy-id" \
  --env.type=metaworld \
  --env.task=medium \
  --eval.batch_size=1 \
  --eval.n_episodes=2
```

This will run episodes and return per-task success rates using the standard Meta-World evaluation keys.

## Async MetaWorld evaluation

You can also decouple simulation from policy inference using the async inference stack:

- `policy_server` runs the policy and returns action chunks.
- `metaworld_client` runs the MetaWorld environments locally, streams observations to the server, and executes returned actions.

Start the async policy server first:

```bash
python -m lerobot.async_inference.policy_server \
  --host=0.0.0.0 \
  --port=8080 \
  --fps=80
```

Then launch the MetaWorld async client:

```bash
python -m lerobot.async_inference.metaworld_client \
  --policy.path=/path/to/checkpoint/pretrained_model \
  --server_address=127.0.0.1:8080 \
  --env.task=medium \
  --env.fps=80 \
  --actions_per_chunk=15 \
  --chunk_size_threshold=0.5 \
  --n_episodes=2
```

Notes:

- `--env.task` accepts either a difficulty group (`medium`, `hard`, ...) or a comma-separated list of task names.
  Built-in groups are `all`, `easy`, `medium`, `hard`, and `very_hard`.
- `--env.fps` controls the local simulation control rate.
- `--actions_per_chunk` is the maximum number of actions the server sends back per request. The effective value is capped by the policy's own `chunk_size`.
- `--chunk_size_threshold` controls when the client starts requesting the next chunk. With `actions_per_chunk=15` and `chunk_size_threshold=0.5`, replanning starts when the local queue shrinks to about 7 remaining actions.
- For the cleanest timing semantics, keep `policy_server --fps` aligned with `metaworld_client --env.fps`.

### How async replanning works

The MetaWorld async client follows the same control model as `RobotClient`:

1. Execute the current local action queue.
2. Once the queue falls below `chunk_size_threshold`, send a fresh observation.
3. The server keeps only the latest useful observation and may filter out similar ones.
4. When a new chunk arrives, merge it into the local queue and continue execution.

This means it is normal to see more observations sent than actually enqueued for inference. Filtered observations are part of the design and help avoid recomputing nearly identical plans.

### Async CLI reference

#### `metaworld_client` core async parameters

| Parameter | Default | Choices / constraints | Description |
|---|---:|---|---|
| `--server_address` | `localhost:8080` | Any `host:port` | gRPC address of the policy server |
| `--n_episodes` | `1` | Positive integer | Number of episodes to run per task |
| `--seed` | `0` | Integer | Base random seed for evaluation |
| `--actions_per_chunk` | `50` | Positive integer | Maximum number of actions requested from the server per response |
| `--chunk_size_threshold` | `0.5` | Float in `[0, 1]` | Start requesting a new chunk when `queue_size / effective_chunk_size <= threshold` |
| `--aggregate_fn_name` | `weighted_average` | `weighted_average`, `latest_only`, `average`, `conservative` | How overlapping actions are merged locally |
| `--client_device` | `cpu` | Any torch device string | Device used for received action tensors on the client |
| `--debug_visualize_queue_size` | `false` | `true` / `false` | Plot the local action queue size after evaluation |

#### MetaWorld environment parameters

| Parameter | Default | Choices / constraints | Description |
|---|---:|---|---|
| `--env.task` | `metaworld-push-v2` | Difficulty group or task name, comma-separated allowed | Examples: `all`, `medium`, `hard`, `hammer-v3,push-wall-v3` |
| `--env.fps` | `80` | Positive integer | Local simulation control rate |
| `--env.episode_length` | `400` | Positive integer | Maximum environment steps per episode |
| `--env.obs_type` | `pixels_agent_pos` | `pixels`, `pixels_agent_pos` | Observation layout used by the client |
| `--env.render_mode` | `rgb_array` | Effectively `rgb_array` in current setup | MetaWorld render mode |
| `--env.multitask_eval` | `true` | `true` / `false` | MetaWorld multitask evaluation config flag |
| `--env.disable_env_checker` | `true` | `true` / `false` | Gym env checker toggle |
| `--env.max_parallel_tasks` | `1` | Positive integer | Present in config; current async client evaluates tasks sequentially |

#### Policy parameters

`--policy.path` is required. Other `--policy.*` arguments override values stored in the checkpoint config.

| Parameter | Default | Choices / constraints | Description |
|---|---:|---|---|
| `--policy.path` | Required | Local directory or HF repo id | Path to the `pretrained_model` directory or Hub model |
| `--policy.device` | From checkpoint | Any torch device string | Device used to load the policy |
| `--policy.compile_model` | From checkpoint | `true` / `false` | Whether the policy is compiled; async server currently disables this at load time for stability |
| `--policy.dtype` | Policy-specific, e.g. `float32` for `pi0` | Depends on policy | Inference dtype |
| `--policy.chunk_size` | Policy-specific, e.g. `50` for `pi0` | Positive integer | Maximum number of actions the policy can predict in one call |
| `--policy.n_action_steps` | Policy-specific, e.g. `50` for `pi0` | Positive integer, typically `<= chunk_size` | Mainly used by local `select_action()` paths; not the main limiter in async server mode |

For async inference, the effective returned chunk size is:

```text
effective_chunk_size = min(policy.chunk_size, actions_per_chunk)
```

So if your checkpoint has `--policy.chunk_size=15` and you launch the async client with `--actions_per_chunk=50`, the server still returns at most `15` actions per request.

#### `policy_server` parameters

| Parameter | Default | Choices / constraints | Description |
|---|---:|---|---|
| `--host` | `localhost` | Any bind address | Address used by the gRPC server |
| `--port` | `8080` | Integer in `[1, 65535]` | Port used by the gRPC server |
| `--fps` | `30` | Positive integer | Server-side action timestamp spacing; best kept aligned with `--env.fps` |
| `--inference_latency` | `1 / 30 ≈ 0.0333` | Non-negative float | Target inference latency config |
| `--obs_queue_timeout` | `2` | Non-negative float | Timeout while waiting for a new observation |

## Practical tips

- If you care about generalization, run on the full MT50 suite — it’s intentionally challenging and reveals strengths/weaknesses better than a few narrow tasks.
- Use the one-hot task conditioning for multi-task training (MT10 / MT50 conventions) so policies have explicit task context.
- Inspect the dataset task descriptions and the `info["is_success"]` keys when writing post-processing or logging so your success metrics line up with the benchmark.
